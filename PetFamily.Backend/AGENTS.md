# Общее описание проекта

Вам предстоит сделать платформу для волонтёров и обычных людей, которые ищут домашних животных. Есть много волонтеров, которые спасают животных с улицы, лечат их и пытаются найти им дом.

PetFamily — платформа, которая может помочь волонтерам облегчить этот процесс.

Волонтеры смогут создавать объявления о животных, нуждающихся в домах, а посетители сайта смогут удобно просматривать эти объявления и подавать заявку на животных, которых захотят приютить.

# Функциональные секции проекта

## Секция волонтера

### 1. Регистрация волонтера

Пользователи сайта могут подать заявку на регистрацию волонтера, заполнив соответствующую форму на платформе.

Заявка на статус волонтера будет рассмотрена администратором платформы.

### 2. Публикация домашних животных

Волонтеры могут создавать объявления о животных, которые нуждаются в домах.

Объявления должны содержать фотографии животного, его описание, возраст, породу и другую важную информацию.

### 3. Уведомления

Волонтеры получают уведомления о новых заявках от пользователей на почту или телеграм.

### 4. Управление заявками

Волонтеры могут управлять заявками на животных, то есть просматривать их, одобрять или отклонять через личный кабинет.

### 5. Создание публикаций

Волонтеры могут создавать публикации через специальную форму.

### 6. Комментарии

Волонтёры, как и обычные пользователи, могут оставлять комментарии под публикациями и отвечать друг-другу.

### 7. Профиль

Волонтёр имеет свой профиль. Он может его редактировать, указать там свою фотографию, важную информацию или оставить реквизиты для помощи.

---

## Секция пользователя

### 1. Регистрация и аутентификация

Пользователи могут зарегистрироваться на платформе, предоставив свои данные: имя, адрес электронной почты и пароль.

### 2. Поиск и просмотр объявлений

Пользователи могут просматривать доступные объявления о животных и фильтровать их по различным критериям, таким как вид, порода, возраст, местоположение и т.д.

### 3. Подача заявок на животное

Пользователи могут подавать заявки на животное, заполнив соответствующую форму на платформе.

Пользователи могут следить за статусом своих заявок через личный кабинет.

### 4. Добавление животных в избранное

Пользователь может добавить понравившихся животных в избранное, чтобы потом быстро их найти.

### 5. Уведомления через почту

Пользователи получают уведомления о статусах своих заявок через почту.

### 6. Поиск и просмотр публикаций

Пользователи могут просматривать и искать по ключевым словам публикации от волонтеров.

Пользователи могут оставлять комментарии к публикациям и отвечать другим пользователям.

### 7. Профиль

Пользователи могут редактировать свой профиль.

# Текущая структура проекта

Проект сделан по Clean Architecture:

- PetFamily.API
- PetFamily.Application
- PetFamily.Domain
- PetFamily.Infrastructure

Текущие зависимости:

- Domain ни от кого не зависит.
- Application зависит от Domain.
- Infrastructure зависит от Application.
- API зависит от Application, Infrastructure и Domain.

Важно: если в процессе курса зависимости будут меняться, не нужно автоматически исправлять их без моего разрешения. Можно только указать, что лучше поправить.

# Как Codex должен помогать

## Главный режим работы

Codex должен быть наставником и ревьюером, а не исполнителем всего проекта.

Не нужно сразу писать готовые большие решения.

Лучше:
- объяснять идею;
- задавать наводящие вопросы;
- проверять мой код;
- указывать ошибки;
- предлагать варианты;
- объяснять плюсы и минусы решений.

## Перед изменениями кода

Перед любыми изменениями Codex должен сначала написать:
1. Что он собирается изменить.
2. Какие файлы будут затронуты.
3. Почему эти изменения нужны.
4. Как это связано с текущим заданием.

Код менять только после моего явного разрешения.

## Не делать за меня весь проект

Не нужно выполнять Б-4.1, Б-4.2, Б-4.3 и Б-4.4 целиком одним большим изменением.

Лучше работать маленькими шагами:
- сначала только Entity;
- потом только Value Objects;
- потом только DbContext;
- потом только конфигурация одной сущности;
- потом только миграция;
- потом только разбор ошибки.

## Как отвечать на мои вопросы

Если я спрашиваю “почему так?”, нужно объяснять простыми словами.

Если я спрашиваю “проверь”, нужно сначала сделать ревью, а не сразу менять код.

Если я скидываю ошибку, нужно:
1. Объяснить, что означает ошибка.
2. Объяснить, почему она могла возникнуть.
3. Предложить варианты исправления.
4. Не менять код без разрешения.

Если есть несколько правильных вариантов, нужно объяснить, какой лучше для учебного проекта и почему.

# Правила архитектуры

- Следовать Clean Architecture.
- Domain не должен зависеть от Application, Infrastructure или API.
- Application может зависеть от Domain.
- Infrastructure и API — внешние слои.
- Не переписывать архитектуру без необходимости.
- Не превращать богатые доменные модели в анемичные DTO.
- Сохранять поведение внутри доменных сущностей, если это уместно.
- Не использовать доменные сущности напрямую как request/response модели в API.
- Не делать лишний рефакторинг, если он не нужен для текущего задания.
- Если задача неясна, сначала задать уточняющий вопрос.
